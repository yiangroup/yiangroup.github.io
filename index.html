Step 01 Clinical Research Environment & Regulatory Compliance Clinical context, compliance expectations, core documents, data flow, and programming standards.

1. Regulatory Framework & Red Lines

Clinical trial data collection, programming, analysis, and reporting must follow international quality standards and local regulatory requirements. For statistical programmers, compliance is not separate from programming; it shapes how data is collected, transformed, reviewed, and submitted.

1.1 Global Regulatory Framework

  • ICH-GCP E6(R2/R3): Good Clinical Practice is the international ethical and quality standard for designing, conducting, recording, and reporting clinical trials. In programming work, GCP means that data manipulations, analyses, and reporting outputs must protect subject safety and preserve trial integrity.
  • 21 CFR Part 11 and EMA Annex 11: These regulations address electronic records, electronic signatures, system access, audit trails, and data reliability. Programming environments must support controlled access, traceable changes, secure storage, and reproducible execution.
  • Data privacy regulations: HIPAA in the United States and GDPR in the European Union protect patient privacy. Protected Health Information should not appear in analysis-ready programming environments unless specifically permitted and controlled. De-identification, pseudonymization, date handling, and access control are part of responsible data handling.

1.2 Electronic Records: 21 CFR Part 11 & EMA Annex 11

In a digital clinical trial environment, electronic data, source code, execution logs, dataset timestamps, review comments, and electronic signatures are regulated records. The core expectation is that these records must be authentic, reliable, attributable, protected from unauthorized change, and reproducible during an inspection.

  • No manual overwrites: Programmers should not open a SAS dataset, R dataframe, CSV file, or Excel listing and directly alter a data point. Even an obvious typo must be corrected through the proper data management query process, fixed in the source system when appropriate, and then reprocessed through the controlled pipeline.
  • Traceable code changes: Code updates must be tracked in a validated version control or programming environment. A commit message such as "fixed bug" is not enough for regulated work; the change should explain why it was made, what logic or output was affected, and which protocol, SAP, shell, specification, or review finding required the update.
  • Unique user accountability: User IDs and passwords must not be shared. Server audit trails tie logins, file edits, program executions, and approvals to a specific person. Programmers should also lock their workstation whenever they step away, because work performed under their account is attributed to them.
  • Controlled environments: Production data and production outputs are usually protected by read-only access, release controls, and validated execution paths. Programmers should not install unapproved R, Python, or SAS components on production systems or bypass Dev-to-Prod promotion workflows.
  • Clean, retained logs: SAS logs, R outputs, validation reports, and execution metadata are inspection evidence. Errors, warnings, automatic type conversions, uninitialized variables, and unexplained record count changes should be investigated and resolved rather than hidden or deleted.

1.3 Data Privacy: HIPAA, GDPR, and Patient-Level Data

Statistical programmers are one of the final safeguards against privacy leakage. HIPAA, GDPR, and sponsor policies require patient-level data to be minimized, pseudonymized, access-controlled, and transferred only through approved channels.

  • Remove direct identifiers: Analysis-ready datasets should not contain patient names, full addresses, phone numbers, email addresses, national identifiers, medical record numbers, or other direct identifiers unless a specific controlled process requires them.
  • Use pseudonymized subject keys: Subjects should be tracked with consistent study identifiers such as USUBJID, commonly formed from protocol, site, and subject number. The same pseudonymized key must remain consistent across SDTM, ADaM, listings, and reconciliation outputs.
  • Handle dates carefully: Exact birth dates and full calendar dates can become indirect identifiers, especially in rare disease or small studies. Programmers may need to derive age, study day, or shifted dates according to the protocol, data transfer agreement, SAP, and privacy requirements, then remove the unnecessary high-risk source date.
  • Audit free text: Verbatim fields and comment fields can contain accidental identifiers, such as a patient name inside an adverse event term or a protocol deviation description. These fields should be reviewed, queried, masked, or redacted through a controlled process before delivery.
  • Keep data inside approved systems: Patient-level SAS datasets, XPT files, CSVs, Excel listings, screenshots, and extracts should not be saved to personal drives, unsecured local folders, external drives, personal email, or chat attachments. Use secure project directories and permission-controlled file links.

1.4 Where Data Standards Fit

A new programmer does not need to learn every data standard on the first day. The first idea is simpler: clinical trial data cannot stay as a pile of site forms, EDC exports, vendor spreadsheets, and one-off file layouts. Before analysis and submission, the data must be reorganized into a standard path so the study team and regulators can follow where each value came from and how it was used.

The standard language behind that path is CDISC. CDISC, the Clinical Data Interchange Standards Consortium, is a global, non-profit organization that defines data standards for clinical research. Think of CDISC as the universal grammar for clinical trials: because of it, a reviewer at the FDA or EMA can open a submission dataset and know where to find a subject's age, adverse events, lab results, exposure records, and analysis variables.

Raw Data Collected
Site forms, EDC exports, and vendor files.
CDASH
Input standard for cleaner CRF and EDC collection.
SDTM
Tabulation standard showing what happened to the subject.
ADaM
Analysis-ready datasets with derivations and population flags.
TFLs
Final tables, listings, and figures for reporting.
  • CDASH is the input standard: It helps design CRF and EDC fields so data is collected consistently from the beginning.
  • SDTM is the tabulation standard: It organizes collected clinical data into standard domains such as DM for demographics, AE for adverse events, LB for labs, and VS for vital signs. At this stage, the goal is to represent what happened to the subject, not to perform statistical analysis.
  • ADaM is the analysis standard: It uses SDTM and SAP rules to build analysis-ready datasets, including baseline values, change from baseline, imputation rules, analysis visits, and population flags. Final TFLs are generated from ADaM datasets.

This pipeline explains why private shortcuts are forbidden. A shortcut means bypassing part of the approved chain, such as pulling raw EDC data directly into a final table because it feels faster. That may save time today, but it breaks the ability to trace a final number back to ADaM, SDTM, and the original source record.

We will introduce the detailed programming specifications for CDASH, SDTM, and ADaM later in the data mapping lessons. For this first compliance lesson, remember the practical rule: respect the pipeline, document every derivation, and protect the chain of custody from source data to final tables.

1.5 Red Lines: Compliance Boundaries

The following red lines are practical boundaries for clinical programming. Crossing them can compromise data integrity, patient privacy, study validity, or regulatory credibility.

  • ALCOA+ data integrity: Data and programming actions should be attributable, legible, contemporaneous, original, accurate, complete, consistent, enduring, and available. Untraceable data changes, back-dated sign-offs, and undocumented fixes are not acceptable.
  • No hardcoded source data fixes: Programmers should not directly correct subject data inside SAS, R, or Python code, such as forcing a single subject's value to a desired result. Source data corrections should go through appropriate data management processes and audit trails.
  • Blinding integrity: In blinded studies, programmers should not access randomization or unblinded treatment information unless formally assigned to an unblinded role. Premature or unauthorized blind-breaking analyses can damage the interpretability of the trial.
  • Patient privacy: Names, initials, national IDs, exact addresses, and other direct identifiers should not be included in SDTM, ADaM, TFLs, or general analysis environments unless a controlled and approved process requires it.
  • Independent QC: Primary and QC programming should remain independent where required. Copying production code into QC to force a match defeats the purpose of validation and can hide logic errors.
  • Validated tools and audit trails: Submission-critical work should use approved and validated systems, macros, packages, and workflows. Logs should not be deleted or edited to hide errors, warnings, or important notes.

2. Bridge to the Study Documents

The compliance rules above may sound abstract at first. In daily programming, they become much more concrete because programmers do not make rules from memory. They check approved study documents, then turn those instructions into traceable code.

For the first lesson, only remember the basic job of each document. The next lesson will show how to read them in detail.

  • Protocol: Explains what the trial is trying to study, who can join, what visits happen, and which assessments are collected.
  • SAP: Explains how the study will be analyzed, including populations, baseline rules, endpoints, missing data rules, and statistical methods.
  • TLF shells: Show what the final tables, listings, and figures should look like, including titles, rows, columns, and footnotes.
  • aCRF and vendor specs: Help programmers find where source data came from, whether it was entered in the EDC or transferred from an outside vendor such as a lab or ECG provider.

This is the key transition into Step 02: if Step 01 teaches what not to violate, Step 02 teaches where to find the approved instructions before writing code.

4. SOPs & Coding Standards

In a regulated industry, code is not just a personal tool. It is a collaborative, permanent corporate asset. If a programmer goes on vacation, leaves the company, or joins another study, a teammate should be able to open the program and understand the purpose, inputs, outputs, assumptions, and review history within minutes.

  • SOPs control the process: Standard Operating Procedures define how study folders are created, who can move files from Development to Production, how review comments are documented, how data issues are escalated, and how final deliverables are archived.
  • Training sign-off matters: Before touching active study data or production programs, new hires usually complete a training matrix and sign off that they have read and understood the required programming SOPs.
  • Coding standards control the script: Standards cover headers, indentation, naming conventions, comments, log checks, output locations, and version history so code remains readable, reproducible, and reviewable.

A compliant program usually begins with a required header block. This is not decoration; it tells the reviewer what the program does, which data it uses, what it creates, who wrote it, and why later changes were made.

SAS Program Header Example .sas 
/*****************************************************************************
* Company Name: YIAN GROUP Training
* Study Protocol: ABC-123
* Program Name: adae.sas
* Purpose: Create the Adverse Event Analysis Dataset (ADAE)
* Inputs: sdtm.ae, sdtm.dm, adam.adsl
* Outputs: adam.adae
*-----------------------------------------------------------------------------
* Author: John Doe (j.doe@company.com)
* Creation Date: 2026-06-28
*-----------------------------------------------------------------------------
* Modification History:
* Date        Author       Reference      Description of Change
* ----------  -----------  -------------  ------------------------------------
* 2026-07-15  Jane Smith   Ticket #442    Updated missing onset date imputation
*                                         per SAP Amendment 2.
*****************************************************************************/

Practical rule: never start functional SAS, R, or Python code on line 1. First give the reviewer a map of the program.

4.1 Readable Code Habits
  • Use consistent layout: Follow the team standard for indentation, spacing, dataset names, variable names, and section breaks. A reviewer should not need to decode your personal style.
  • Remove dead code: Do not leave large blocks of commented-out experimental code in final production programs. Version control keeps history; production programs should show the current approved logic.
  • Comment the reason, not the obvious action: Comments should explain why logic exists, especially for protocol exceptions, SAP rules, data anomalies, or sponsor decisions.
SAS Comment Examples Good vs Bad 
/* Bad: repeats the code without explaining the reason. */
/* Substringing the variable */
age_num = input(substr(ageres, 1, 2), best.);

/* Better: explains the study rule and document source. */
/* Remove the "YEARS" suffix from AGERES before numeric conversion,
   per ADaM specification page 12. */
age_num = input(scan(ageres, 1, " "), best.);
4.2 Log Checking: The Ultimate Defense

A program that runs is not automatically correct. The execution log is often the first place where data integrity problems appear. Before submitting work for review, programmers should run the team log-check utility or inspect logs line by line.

Critical Keywords to Search and Resolve
In SAS Environments In R / Python Environments Action Required by the Programmer
ERROR: Error: Critical failure. The program stopped executing. Must be fixed immediately.
WARNING: Warning: Potential data corruption, such as dropped variables, length mismatches, or failed calculations. Must be resolved.
uninitialized NaN / NA creation A variable was called but does not exist, or a calculation created missing values unexpectedly. Check spelling, input structure, and upstream data changes.
w.d format too small Numerical truncation or coercion warnings Data values may be clipped, rounded, or converted incorrectly on export. Use explicit formats and verify displayed values.
MERGE statement has repeats Many-to-many merge or join warning Unexpected data multiplication may have occurred. Confirm unique key variables before merging and compare record counts before and after the join.

4.3 Independent QC and Compare

Regulated programming relies on independent validation. The production programmer writes the main program, and a QC programmer independently interprets the same SAP and specifications. The goal is not to copy the production code; the goal is to catch misunderstanding before delivery.

Production deliverables should remain as produced and archived. In QC, compare directly against the production dataset and prepare only the independent QC dataset if sorting or key alignment is needed.

Production Programmer
Creates the production dataset or output from approved specifications.
QC Programmer
Builds an independent QC version from the same documents.
Automated Compare
Compares records, variables, labels, formats, values, and types.
Resolution
Any mismatch is investigated, documented, corrected, and rerun.
SAS PROC COMPARE Example QC 
/* Do not modify, overwrite, or pre-sort the production deliverable.
   Compare directly against adam.adae. */
data qc_adae;
  set qc.adae;
run;

proc sort data=qc_adae;
  by usubjid aeseq paramcd avisitn;
run;

proc compare base=adam.adae
             compare=qc_adae
             criterion=1e-10
             listall;
  id usubjid aeseq paramcd avisitn;
run;
Onboarding Takeaway

SOPs protect the company from regulatory findings. Coding standards protect your teammates from unreadable or unreproducible code. Write every program as if the next person opening it must defend it during an audit.

Step 02 Clinical Trial Documents Programmers Must Read Protocol, SAP, TLF shells, annotated CRF, data transfer specifications, and Define.xml.

1. Why Documents Come Before Programming

Statistical programming is specification-driven work. Every dataset, derived variable, output title, footnote, population flag, and quality-control check is governed by approved study documents. These documents are the inputs, rules, and blueprints for compliant programming.

A programmer should not code from memory or from an informal chat message. When a rule is unclear, the correct response is to identify the controlling document, record the question, and confirm the decision with the statistician, data manager, or study team.

2. Document Dependency Map

SAP
Analysis populations, estimands, baseline rules, missing data rules, and statistical methods.
SDTM / ADaM Specs
Excel-level programming details: variables, labels, types, origins, derivations, and algorithms.
aCRF / DTS / Dictionaries
Source fields, vendor transfer rules, MedDRA, WHODrug, and coded safety terms.

SAS or R programs create standardized datasets and final outputs.
P21 Report

Define.xml + SDRG / ADRG

4. Clinical Study Protocol: The Master Blueprint

The protocol is the legal and operational blueprint of the clinical trial. It defines why the study is being run, who can enter, what treatment they receive, which assessments are collected, and what clinical questions the study is meant to answer.

  • Schedule of Activities: Use the SoA to understand which assessments are expected at Screening, Baseline, treatment visits, follow-up, and End of Study. It is the first source for visit timing and expected data.
  • Objectives and endpoints: Read the primary, secondary, and exploratory endpoints before building endpoint variables. The endpoint wording often determines the analysis value, baseline definition, and target visit.
  • Eligibility rules: Inclusion and exclusion criteria support eligibility checks, protocol deviation review, and population definitions.
  • Programming impact: The protocol drives visit-window logic, expected assessment checks, treatment period definitions, unscheduled visit handling, and questions such as whether an ECG, lab, or vital sign record is expected at a specific visit.

5. Statistical Analysis Plan: The Daily Programming Rulebook

The SAP turns the clinical intent of the protocol into executable analysis rules. For ADaM and TFL programming, this is usually the most frequently opened document.

  • Analysis populations: Translate population definitions into precise ADSL flags such as SAFFL, FASFL, ITTFL, PPSFL, or other study-specific flags. Do not infer a population from the shell alone.
  • Baseline definitions: Implement the exact SAP rule, including date and time logic. If baseline is the last non-missing value before first dose, a Day 1 post-dose value must not accidentally become baseline.
  • Missing data and imputation: Follow the specified approach for missing efficacy, safety, dates, severity, relationship, or endpoint values. LOCF, multiple imputation, non-responder imputation, and partial date imputation are not interchangeable.
  • Derived variables and windows: Use SAP rules for study day, change from baseline, BMI, endpoint derivations, visit windows, analysis visit assignment, censoring, and sensitivity analyses.

6. TLF Mock Shells: The Target Output Design

Mock shells show what the final tables, listings, and figures should look like. They are not just formatting examples; they contain programming instructions in titles, subtitles, column headers, row labels, and footnotes.

  • Titles and headers: Confirm table number, endpoint, visit, treatment groups, population, and whether the output is descriptive, inferential, safety, efficacy, or listing-based.
  • Rows and columns: Match grouping, sorting, indentation, decimal precision, statistics, denominator rules, and treatment comparisons.
  • Footnotes: Read every footnote before coding. Footnotes often define source datasets, analysis populations, statistical models, date cutoffs, denominator logic, and abbreviation meanings.
  • Programming impact: Final output code should match the shell layout closely, but the source and logic must still be reconciled against the SAP and specifications.

7. SDTM / ADaM Programming Specifications: The Code Translator

The SAP gives the statistical rule, but the programming specification turns that rule into dataset-level and variable-level instructions. In many projects, the spec is a large Excel workbook that programmers open constantly while writing SDTM, ADaM, and TFL code.

  • Metadata columns: Learn to read variable name, label, type, length, format, controlled terminology, role, origin, and comments.
  • Derivation or algorithm column: This is the direct bridge to code. It should explain how each variable is assigned, copied, merged, derived, imputed, flagged, or populated conditionally.
  • Spec versus SAP: The SAP defines the analysis intent. The spec translates that intent into implementable dataset logic. If they conflict, stop and raise a question instead of choosing one silently.
  • Programming impact: Do not build production code from an unfinished or unapproved spec unless the team has explicitly authorized that development stage. Draft specs often change and can cause avoidable rework.

8. Annotated CRF: The Source Data Map

The annotated CRF is the bridge between what the site entered in the EDC and how the data is represented in SDTM. It helps programmers understand where each source field came from and how it should be interpreted.

  • Domain markings: Page and field annotations point to domains such as DM, AE, CM, LB, VS, ECG, or exposure-related domains.
  • Variable annotations: Field-level labels indicate target SDTM variables or mapping clues, such as DM.SEX, VSORRES, VSPOS, or AETERM.
  • Origin metadata: Determine whether a value is collected on the CRF, derived by the system, assigned by the sponsor, or imported from an external source.
  • Programming impact: If a raw variable is not annotated or an annotated CRF field has no raw data, flag the discrepancy. Do not silently map unmapped data or leave collected data behind without explanation.

9. Data Transfer Specifications: Vendor Data Rules

Many important data streams do not come directly from the EDC. Central labs, ECG vendors, PK labs, imaging vendors, ePRO vendors, and biomarker labs often transfer data using a Data Transfer Specification.

  • File structure: Check file names, file type, delimiter, encoding, expected worksheets, header rows, required columns, and transfer frequency.
  • Merge keys: Confirm subject ID, site, visit, sample ID, accession number, specimen collection date/time, and any vendor-specific identifiers used for reconciliation.
  • Units and codelists: Validate units, result codes, normal ranges, flags, test names, date formats, missing value conventions, and version-controlled dictionaries.
  • Programming impact: Incoming vendor files should be checked against the DTS before production processing. A changed column name or date format can silently corrupt study day, visit assignment, or merge results.

10. MedDRA and WHODrug: External Coding Dictionaries

Safety programming depends heavily on coded medical terminology. Site-entered adverse events, medical history terms, and concomitant medications often begin as verbatim text, then Data Management codes them using controlled external dictionaries before analysis.

  • MedDRA: Used for adverse events and medical history. It provides hierarchy such as System Organ Class and Preferred Term, which often drive safety table grouping and sorting.
  • WHODrug: Used for concomitant medications and includes drug names, ingredients, and ATC classifications.
  • Programming impact: ADAE, ADCM, and safety TFLs frequently rely on coded variables such as AEBODSYS, AEDECOD, CMDECOD, and ATC classes. Frequency counts and grouping logic should use the approved coded terms, not raw verbatim text.
  • Review habit: If dictionary-coded fields are missing, inconsistent, or unexpectedly changed between transfers, raise the issue with Data Management instead of recoding terms manually in analysis code.

11. Define.xml and Submission Metadata

Define.xml explains the submitted datasets to reviewers. It documents dataset purpose, variable labels, variable attributes, origins, derivations, value-level metadata, controlled terminology, and links to reviewer guides or external codelists.

  • Use it as a traceability check: The final SDTM and ADaM datasets should agree with the metadata that define.xml describes.
  • Check variable-level details: Labels, types, lengths, codelists, origins, and derivations should be consistent with specifications and actual datasets.
  • Programming impact: If a submitted variable is derived, the derivation should be understandable from the metadata and traceable to source records or documented rules.

12. SDRG and ADRG: Reviewer Guides

Define.xml is metadata, but it does not tell the whole story. Reviewer guides explain the study data package in plain language. The SDRG explains SDTM data, while the ADRG explains ADaM data and analysis-specific decisions.

  • SDRG: The Study Data Reviewer's Guide explains SDTM package structure, known data issues, conformance notes, special study design considerations, and unresolved validation findings.
  • ADRG: The Analysis Data Reviewer's Guide explains ADaM datasets, analysis derivations, traceability, departures from standard expectations, and how analysis results should be interpreted.
  • Programmer responsibility: Programmers are often major contributors to the ADRG. If code intentionally handles unusual data, deviates from a common standard because of study design, or leaves a validation issue unresolved with justification, the reviewer guide must explain it.
  • Practical rule: Anything that would make a reviewer ask "Why did the data look like this?" probably belongs in SDRG, ADRG, or a documented response trail.

13. Pinnacle 21 Validation Report: The Automated Rule Check

After datasets and metadata are built, validation tools such as Pinnacle 21 are used to scan SDTM, ADaM, and Define.xml against conformance rules. The resulting report is a core review document for programmers, not an afterthought.

  • Severity levels: Learn to triage Reject, Error, Warning, and Notice findings. High-severity findings can block submission readiness, while lower-severity findings still need review and disposition.
  • Resolution loop: Read the P21 rule, identify the affected dataset or variable, check the spec and code, fix the program when appropriate, rerun validation, and document any finding that remains by design.
  • Connection to reviewer guides: If a finding cannot or should not be fixed because it reflects the protocol, data reality, or approved study design, the explanation should be captured in SDRG or ADRG.

14. How to Resolve Document Questions

A good programmer does not guess when documents seem unclear. The usual approach is to identify the controlling document, record the question clearly, and confirm with the statistician, data manager, or study team as appropriate.

  • Clinical timing: Start with the Protocol and Schedule of Assessments.
  • Analysis rule: Start with the SAP and programming specifications.
  • Output layout: Start with TLF shells and output specifications.
  • Source field: Start with annotated CRF, raw data, and data transfer specifications.
  • Safety coding: Check coded terms, MedDRA, WHODrug, and Data Management coding decisions.
  • Submission metadata: Check Define.xml, SDRG, ADRG, and the study metadata package.
  • Validation finding: Check the P21 report, rule details, specification, code, and reviewer guide explanation.

Challenge Scenario 1: The Dual-Visit Windowing Trap (BDS)

Subject 101-002-015 missed the official Week 12 clinic visit. The raw database shows two unscheduled safety checks: Study Day 80 and Study Day 94. Both contain seated systolic blood pressure, and both fall inside the Week 12 window of Study Day 78 to 98.

  1. Why can you not include both records in the Week 12 table?
  2. How should the code choose which one record becomes the analytical Week 12 value in ADVS?
  3. Which document contains the exact tie-breaking rule?
Trainer Blueprint

Look up the SAP section on windowing algorithms and handling multiple assessments. The usual rule is to select the record closest to the target study day; if records are equidistant, use the later record.

  • Week 12 target day is Day 84.
  • Day 80 distance is |80 - 84| = 4 days.
  • Day 94 distance is |94 - 84| = 10 days.
  • Day 80 is selected for Week 12.

data advs_window;
  set advs_candidates;
  target = 84;
  dist = abs(ady - target);
run;

proc sort data=advs_window;
  by usubjid avisitn dist descending ady;
run;

data advs_week12;
  set advs_window;
  by usubjid avisitn;
  if first.avisitn;
run;

Challenge Scenario 3: The Timing Is Everything Baseline Puzzle

Subject 101-001-009 has two vital signs records on Study Day 1, the first-dose day. Record A has systolic BP 142 mmHg at 08:15 AM. Record B has systolic BP 135 mmHg at 11:30 AM. Exposure data shows the first dose was taken at 09:00 AM. The mock shell requires change from baseline.

  1. Which record is the true baseline value?
  2. What happens if the wrong record is used?
  3. If the raw EDC time is missing, which fallback rule must the program use, and which document authorizes it?
Trainer Blueprint

Look up the SAP baseline definition and confirm source time variables in the annotated CRF or data specifications. Record A is baseline because it occurs before the 09:00 AM dose. Record B is post-dose and using it would corrupt the change-from-baseline calculation.

  • Use date-time logic such as VSDTM < EXDTM when both timestamps are available.
  • If time is missing, use the SAP's missing-time fallback rule.
  • Many SAPs state that a Day 1 missing-time assessment is assumed pre-dose unless explicitly marked post-dose.
  • The program should also use VISIT, VSEVAL, or another clinical-intent variable when available.
SAS Baseline Blueprint ADVS 
/* Example logic only. Final missing-time rules must follow the SAP. */
if not missing(vsdtm) and not missing(exdtm) then pre_dose = (vsdtm < exdtm);
else if ady = 1 and missing(vstm) and upcase(visit) = "BASELINE" then pre_dose = 1;
else pre_dose = 0;

if pre_dose then ablfl = "Y";

The first SDTM task is not to copy raw columns. It is to decide each variable's clinical role according to the SDTMIG.

3. Split Domains and SUPP-- Architecture

Real studies can produce very large datasets. SDTM supports standard ways to manage physical dataset size and nonstandard collected variables without breaking the model.

  • Split domains: If a domain such as QS or LB becomes too large for transport or operational limits, it may be physically split into datasets such as QSPH and QSSL. The parent domain remains the conceptual source.
  • Supplemental qualifiers: If a valuable collected variable has no standard variable slot in the parent domain, do not add a nonstandard column to AE, CM, VS, or another domain. Place it in the appropriate SUPP-- dataset, such as SUPPAE.
  • SUPP-- structure: Programmers must understand key columns such as STUDYID, RDOMAIN, USUBJID, IDVAR, IDVARVAL, QNAM, QLABEL, and QVAL, and know how to transpose between flat source structures and vertical SUPP-- records.

4. Controlled Terminology and Cleaning

Matching the variable name is not enough. The values must also be normalized so the dataset can be reviewed, validated, and reused downstream.

Task What the Programmer Does Common Trap

ISO 8601 date handling Convert dates to --DTC format while preserving partial dates such as 2026-06. Imputing missing day or month in SDTM instead of waiting for ADaM SAP rules.

4.1 SAS Template: Mapping Raw Date Components to --DTC

Modern EDC systems often store dates as separate year, month, day, hour, and minute components. SDTM --DTC variables are character variables, so programmers must build ISO 8601 strings carefully while preserving partial dates.

SAS ISO 8601 Date Template --DTC 
/*****************************************************************************
* Purpose: Build an ISO 8601 compliant SDTM --DTC value from raw eCRF
*          date/time components.
* Handles: Complete dates, partial dates, invalid component combinations,
*          and optional time stamps.
*****************************************************************************/

data sdtm_ae(drop=_:);
  set raw_ae;
  length AEDTC $19;
  call missing(AEDTC);

  length _yyyy $4 _mm _dd _hh _ii $2 _datepart $10 _timepart $5;
  call missing(_yyyy, _mm, _dd, _hh, _ii, _datepart, _timepart);

  /* Year is the anchor. If year is missing, the date is unknown. */
  if not missing(raw_yr) then do;
    _yyyy = put(raw_yr, z4.);

    if not missing(raw_mo) and 1 <= raw_mo <= 12 then do;
      _mm = put(raw_mo, z2.);

      if not missing(raw_dy) and 1 <= raw_dy <= 31 then do;
        _dd = put(raw_dy, z2.);
        _datepart = catx("-", _yyyy, _mm, _dd);

        if not missing(raw_hr) and 0 <= raw_hr <= 23 and
           not missing(raw_mn) and 0 <= raw_mn <= 59 then do;
          _hh = put(raw_hr, z2.);
          _ii = put(raw_mn, z2.);
          _timepart = catx(":", _hh, _ii);
          AEDTC = cats(_datepart, "T", _timepart);
        end;
        else AEDTC = _datepart;
      end;
      else AEDTC = catx("-", _yyyy, _mm);       /* YYYY-MM */
    end;
    else AEDTC = _yyyy;                         /* YYYY */
  end;
  else AEDTC = "";
run;
Review Rule Why It Matters Common Mistake
Use z2. and z4. formats ISO 8601 requires two-digit months and days, such as 2026-01-05. Using a simple PUT format that creates 2026-1-5.
Use the T separator for time A full datetime should look like 2026-06-28T14:30. Joining date and time with a blank, or appending time to a partial date.
Preserve partial dates in SDTM SDTM should keep incomplete dates such as 2026 or 2026-06 when the site did not provide full detail. Guessing missing day or month in SDTM instead of leaving imputation to ADaM rules.
Reject impossible combinations A day without a month cannot be placed on a valid timeline. Creating invalid strings such as 2026--12; the safer output is 2026.
Step 05 Introduction to ADaM Why analysis datasets are built from SDTM and how variables such as baseline, change, and flags support analysis.

1. Why ADaM Exists

SDTM is designed for standardized tabulation, not always for direct analysis. ADaM datasets are analysis-ready datasets created to support the Statistical Analysis Plan and the tables, listings, and figures for a study.

  • Traceability: An analysis value should be traceable back to source and SDTM data.
  • Reproducibility: Another programmer should be able to understand how the result was derived.
  • Analysis readiness: Common derivations are prepared once and reused across outputs.

2. Common ADaM Datasets

  • ADSL: One record per subject, with treatment, population flags, dates, and key subject-level variables.
  • ADVS: Analysis dataset for vital signs, often derived from SDTM VS.
  • ADAE: Analysis dataset for adverse events, often derived from SDTM AE and subject-level treatment information.

Step 06 TLF Basics How tables, listings, and figures differ, and how shells guide statistical programming outputs.

1. What TLF Means

TLF stands for Tables, Listings, and Figures. These outputs summarize and display clinical trial results for clinical study reports, review packages, and internal study interpretation.

  • Tables: Summarized results, such as counts, percentages, means, medians, and confidence intervals.
  • Listings: Subject-level details, often used to review individual data records.
  • Figures: Visual displays such as line plots, forest plots, or Kaplan-Meier curves.

2. What a Shell Provides

A shell is a mock output layout. It tells the programmer what the final output should look like before real numbers are produced.

  • Title, numbering, and population.
  • Rows, columns, treatment groups, and visit structure.
  • Footnotes, abbreviations, and statistical methods.
  • Which dataset and variables are expected to support the output.

Step 07 Programming Workflow & QC Mindset How production programming, independent QC, log review, and documentation help protect analysis quality.

1. The Basic Programming Workflow

Clinical statistical programming is not only about producing code that runs. The work must be organized, reviewable, and reproducible.

  • Read specifications: Review the protocol, SAP, shells, and programming specifications.
  • Build production code: Create datasets or outputs according to approved requirements.
  • Run checks: Review logs, warnings, notes, record counts, and unexpected values.
  • Perform QC: Use independent programming or structured review to confirm the result.
  • Resolve comments: Document decisions and update code or outputs as needed.

2. What QC Is Looking For

QC is a mindset. The goal is not simply to match numbers, but to ask whether the result is supported by the specification and the data.

  • Are the correct subjects included?
  • Are visits, dates, and treatment periods handled correctly?
  • Are missing values, units, and duplicate records handled as specified?
  • Do output titles, footnotes, and formats match the shell?
  • Does the program log show warnings, errors, or concerning notes?

3. Program Headers and Traceability

A program header helps another reviewer understand what the program does and when it changed. It usually includes the program name, purpose, input datasets, output files, author, date, and change history.

Header Item Why It Matters
Purpose Explains what the program is intended to create.
Inputs and Outputs Supports traceability and reruns.
Change History Documents updates made after review or specification changes.

Practice Check

  1. Why is a clean log not enough by itself?
  2. What is one reason QC should be independent when possible?
  3. What should be documented when a program changes?
Answer Key
  1. A clean log does not prove the logic matches the specification.
  2. Independent QC can catch assumptions or logic errors made by the production programmer.
  3. The reason for the change, date, author, and affected output or dataset should be documented.
