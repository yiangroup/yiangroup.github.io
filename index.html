Step 01 Clinical Research Environment & Regulatory Compliance Clinical context, compliance expectations, core documents, data flow, and programming standards.

1. Regulatory Framework & Red Lines

Clinical trials involve patient safety, data integrity, and evidence used for regulatory decision-making. For that reason, the work must follow strict quality and compliance expectations.

  • ICH-GCP: Good Clinical Practice is the international ethical and quality standard for designing, conducting, recording, and reporting clinical trials. A programmer should understand the importance of ALCOA+ data principles: attributable, legible, contemporaneous, original, accurate, and complete.
  • 21 CFR Part 11: This FDA regulation addresses electronic records and electronic signatures. In a regulated environment, code changes, data access, execution history, and approvals may be traceable through audit trails and system controls.

2. Core Clinical Documents

Statistical programmers do not program from assumptions. They program from defined study specifications and approved documents.

  • Protocol: The blueprint of the clinical trial. It defines the study objectives, design, treatment schedule, visit structure, endpoints, and inclusion or exclusion criteria.
  • Statistical Analysis Plan: The analysis guide for the study. It describes statistical methods, derivation rules, analysis populations, endpoint definitions, and the tables, listings, and figures required for delivery.

3. End-to-End Clinical Data Flow

Before working with datasets, a programmer should understand how clinical data moves from patient visits to analysis outputs.

Clinical Site / Hospital
Doctors and site staff collect patient data during study visits.
eCRF / EDC
Patient data is entered into an electronic data capture system.
Raw Datasets
Data management teams clean, query, and export source data.
SDTM Datasets
Raw data is standardized according to CDISC SDTM structures.
ADaM Datasets
Analysis-ready datasets are derived for statistical analysis.
Tables, Listings, and Figures
Final outputs support clinical study reporting and submission packages.

4. SOPs & Coding Standards

Regulated programming teams rely on Standard Operating Procedures and coding standards to keep work consistent, traceable, and transferable across projects.

  • SOP compliance: Programmers follow documented workflows for data access, production programming, quality control, validation, and documentation.
  • Coding standards: Code should be readable, maintainable, and easy for another programmer to review. Program headers, version history, naming conventions, and standard macros are often part of the programming workflow.

Step 02 Clinical Trial Documents Programmers Must Read Protocol, SAP, TLF shells, annotated CRF, data transfer specifications, and Define.xml.

1. Why Documents Come Before Programming

Statistical programming is specification-driven work. Before writing code, the statistician and programmer need to understand what each document controls, where definitions come from, and which source should be used when documents appear to overlap.

2. Core Documents and What to Look For

Document Statistician Focus Programmer Focus
Protocol Study objectives, estimands or endpoints, visit schedule, populations, and key clinical rules. Visit structure, treatment arms, endpoint timing, inclusion/exclusion logic, and assessment schedule.
SAP Analysis populations, endpoint definitions, statistical methods, missing data handling, and sensitivity analyses. Derivation rules, population flags, baseline rules, visit windows, output requirements, and analysis variables.
TLF Shells Output intent, population, endpoint, treatment comparison, statistics, and footnote interpretation. Output layout, row and column structure, formats, titles, footnotes, and QC targets.
Annotated CRF How clinical concepts are collected and where source evidence originates. Source fields, CRF page names, raw variable origin, and SDTM domain mapping clues.
Data Transfer Spec Expected vendor data content, transfer frequency, and scope of externally supplied data. File names, formats, key variables, codelists, units, date formats, and reconciliation checks.
Define.xml Submission metadata, dataset purpose, controlled terminology, and reviewer traceability. Dataset labels, variable attributes, origins, derivations, value-level metadata, and links to codelists.

3. How to Resolve Document Questions

A good programmer does not guess when documents seem unclear. The usual approach is to identify the controlling document, record the question clearly, and confirm with the statistician, data manager, or study team as appropriate.

  • Clinical timing: Start with the Protocol and Schedule of Assessments.
  • Analysis rule: Start with the SAP and programming specifications.
  • Output layout: Start with TLF shells and output specifications.
  • Source field: Start with annotated CRF, raw data, and data transfer specifications.
  • Submission metadata: Check Define.xml and the study metadata package.

Step 03 From eCRF to SDTM How source fields collected on the eCRF are cleaned, exported, and mapped into standardized SDTM domains.

1. Start With SDTM Domain Classes

A strong SDTM mapping process starts by asking what kind of information the raw data represents. SDTM is organized around general observation classes, so a programmer should classify the data first, then choose the specific domain.

Class Main Question Example Domains Common Source
Special Purpose Who is the subject and how is the study organized? DM, CO, SE, SV Enrollment, demographics, visits, study design metadata
Trial Design What was planned before subjects entered the trial? TA, TE, TV, TI, TS Protocol, study design specifications
Interventions What treatment, medication, or action was taken? EX, CM, PR Exposure, concomitant medication, procedure forms
Events What happened to the subject? AE, MH, DS Adverse event, medical history, disposition forms
Findings What was measured, observed, or tested? VS, LB, EG, PE, QS Vital signs, labs, ECG, exams, questionnaires
Findings About What additional finding describes another record? FA Supplementary assessments about events or interventions

2. From eCRF to Raw Dataset to SDTM

The eCRF is designed for data entry. Raw datasets are usually exported in a form-oriented structure. SDTM is a standardized submission structure. Mapping is the work of translating source fields into the right class, domain, and SDTM variables while preserving traceability.

  • Read the annotated CRF: Identify the form, field labels, and intended domain.
  • Inspect raw exports: Confirm field names, values, units, dates, and missing patterns.
  • Choose the class: Decide whether the data is an intervention, event, finding, or special-purpose record.
  • Map to domain variables: Assign identifiers, topic variables, timing variables, and qualifiers.
  • Check traceability: Ensure reviewers can understand where each SDTM record came from.

3. Special Purpose and Trial Design

These domains provide the structure around the subject and the study. They are not usually simple form-to-form mappings, but they are essential because other domains rely on their identifiers, visits, elements, and reference dates.

Domain Purpose Programmer Focus
DM Subject-level demographics and reference information. USUBJID, treatment arm, sex, age, race, country, RFSTDTC, RFENDTC.
SV Subject visits that occurred. Visit names, visit numbers, actual visit dates, unscheduled visits.
TA / TE / TV Planned arms, elements, and visits. Protocol-defined structure used to interpret actual subject data.

4. Interventions

Interventions describe actions taken for or by the subject. These domains often answer what was administered, taken, or performed, and when it happened.

Domain Typical Raw Data Key Mapping Questions
EX Study drug exposure records. What was administered, dose, unit, route, frequency, start/end date?
CM Concomitant medication form. Medication name, indication, dose, unit, route, start/end date, ongoing flag.
PR Procedure records. Procedure term, date, body system or location, reason performed.

5. Events

Events describe occurrences or incidents. They are not measurements like lab values; they are things that happened to the subject and need timing, category, seriousness, or outcome information.

Domain Typical Raw Data Key Mapping Questions
AE Adverse event form. Reported term, start/end date, severity, seriousness, causality, outcome.
MH Medical history form. Condition term, start/end date or year, ongoing status, body system.
DS Disposition and completion status. Completion, discontinuation reason, death, screen failure, milestone date.

6. Findings

Findings are measurements, tests, observations, or questions with results. This is where programmers often need to reshape data: one raw row with many columns may become many SDTM records, one record per test or assessment.

Domain Typical Raw Data Key Mapping Questions
VS Vital signs fields such as SYSBP, DIABP, PULSE, WEIGHT. What test was measured, result, unit, position, visit, date?
LB Central lab transfer or local lab data. Test code, result, unit, normal range, specimen, fasting status, collection date.
EG ECG parameters. Test code, result, unit, method, interpretation, collection date/time.
QS Questionnaire items and scores. Question code, response, score, version, visit, completion date.

7. Findings About

Findings About is used when the result is a finding about another record rather than a standalone clinical assessment. A common beginner mistake is forcing every extra value into SUPP--. The programmer should first ask whether the value is truly supplemental metadata or whether it is a finding about an event or intervention.

  • Use FA when: The data is a finding that describes another event or intervention.
  • Check the parent: The finding should connect clearly to a parent domain record.
  • Avoid guessing: Confirm study conventions and sponsor standards when FA vs SUPP-- is unclear.

8. Mini Mapping Examples

Raw Form / Field Class Likely Domain Mapping Focus
Con Meds: Medication Name, Dose, Start Date Interventions CM CMTRT, CMDOSE, CMDOSU, CMSTDTC.
Adverse Events: Headache, Moderate, Ongoing Events AE AETERM, AESEV, AEENRF or AEENDTC, seriousness and outcome fields.
Vital Signs: SYSBP=128 mmHg Findings VS VSTESTCD, VSTEST, VSORRES, VSORRESU, VSDTC, VISIT.
Lab: Hemoglobin=13.2 g/dL Findings LB LBTESTCD, LBTEST, LBORRES, LBORRESU, LBDTC, normal range variables.

Suggested Workshop: Class-First SDTM Mapping

For each mock raw item, first choose the SDTM class, then choose the likely domain. This mirrors the way many SDTM mapping discussions are handled in practical programming work.

Mock Raw Item Question
Subject took Metformin 500 mg orally from 2026-03-01 to 2026-04-10. Which class and domain?
Subject reported nausea starting on 2026-03-12, mild severity, not serious. Which class and domain?
Week 8 systolic blood pressure was 128 mmHg while seated. Which class and domain?
Hemoglobin result was 13.2 g/dL with lower normal range 12.0 and upper normal range 16.0. Which class and domain?
Answer Key
  1. Metformin is an Intervention, usually CM.
  2. Nausea is an Event, usually AE.
  3. Systolic blood pressure is a Finding, usually VS.
  4. Hemoglobin is a Finding, usually LB.
Step 04 Introduction to ADaM Why analysis datasets are built from SDTM and how variables such as baseline, change, and flags support analysis.

1. Why ADaM Exists

SDTM is designed for standardized tabulation, not always for direct analysis. ADaM datasets are analysis-ready datasets created to support the Statistical Analysis Plan and the tables, listings, and figures for a study.

  • Traceability: An analysis value should be traceable back to source and SDTM data.
  • Reproducibility: Another programmer should be able to understand how the result was derived.
  • Analysis readiness: Common derivations are prepared once and reused across outputs.

2. Common ADaM Datasets

  • ADSL: One record per subject, with treatment, population flags, dates, and key subject-level variables.
  • ADVS: Analysis dataset for vital signs, often derived from SDTM VS.
  • ADAE: Analysis dataset for adverse events, often derived from SDTM AE and subject-level treatment information.

3. Example ADVS Derivations

Variable Meaning Example
BASE Baseline value before treatment or at the baseline visit. 136
AVAL Analysis value at the selected visit. 128
CHG Change from baseline, usually AVAL - BASE. -8
ANL01FL Analysis flag identifying records included in a planned analysis. Y

Practice Check

  1. If BASE is 136 and AVAL is 128, what is CHG?
  2. Which document usually defines how baseline is selected?
  3. Why is traceability important in ADaM?
Answer Key
  1. CHG is -8.
  2. The SAP usually defines the baseline rule.
  3. Traceability allows reviewers to understand how analysis values were derived from source data.
Step 05 TLF Basics How tables, listings, and figures differ, and how shells guide statistical programming outputs.

1. What TLF Means

TLF stands for Tables, Listings, and Figures. These outputs summarize and display clinical trial results for clinical study reports, review packages, and internal study interpretation.

  • Tables: Summarized results, such as counts, percentages, means, medians, and confidence intervals.
  • Listings: Subject-level details, often used to review individual data records.
  • Figures: Visual displays such as line plots, forest plots, or Kaplan-Meier curves.

2. What a Shell Provides

A shell is a mock output layout. It tells the programmer what the final output should look like before real numbers are produced.

  • Title, numbering, and population.
  • Rows, columns, treatment groups, and visit structure.
  • Footnotes, abbreviations, and statistical methods.
  • Which dataset and variables are expected to support the output.

3. Example Shell Reading

Row Label Placebo YGN-101
n xx xx
Mean (SD) xx.x (xx.x) xx.x (xx.x)
Median xx.x xx.x
Min, Max xx.x, xx.x xx.x, xx.x

Practice Check

  1. Is a listing usually summarized or subject-level?
  2. What does a shell help a programmer confirm before coding?
  3. Why should footnotes be reviewed carefully?
Answer Key
  1. A listing is usually subject-level.
  2. A shell confirms layout, rows, columns, population, statistics, and footnotes.
  3. Footnotes may define important analysis rules, denominators, or interpretation details.
Step 06 Programming Workflow & QC Mindset How production programming, independent QC, log review, and documentation help protect analysis quality.

1. The Basic Programming Workflow

Clinical statistical programming is not only about producing code that runs. The work must be organized, reviewable, and reproducible.

  • Read specifications: Review the protocol, SAP, shells, and programming specifications.
  • Build production code: Create datasets or outputs according to approved requirements.
  • Run checks: Review logs, warnings, notes, record counts, and unexpected values.
  • Perform QC: Use independent programming or structured review to confirm the result.
  • Resolve comments: Document decisions and update code or outputs as needed.

2. What QC Is Looking For

QC is a mindset. The goal is not simply to match numbers, but to ask whether the result is supported by the specification and the data.

  • Are the correct subjects included?
  • Are visits, dates, and treatment periods handled correctly?
  • Are missing values, units, and duplicate records handled as specified?
  • Do output titles, footnotes, and formats match the shell?
  • Does the program log show warnings, errors, or concerning notes?

3. Program Headers and Traceability

A program header helps another reviewer understand what the program does and when it changed. It usually includes the program name, purpose, input datasets, output files, author, date, and change history.

Header Item Why It Matters
Purpose Explains what the program is intended to create.
Inputs and Outputs Supports traceability and reruns.
Change History Documents updates made after review or specification changes.

Practice Check

  1. Why is a clean log not enough by itself?
  2. What is one reason QC should be independent when possible?
  3. What should be documented when a program changes?
Answer Key
  1. A clean log does not prove the logic matches the specification.
  2. Independent QC can catch assumptions or logic errors made by the production programmer.
  3. The reason for the change, date, author, and affected output or dataset should be documented.
